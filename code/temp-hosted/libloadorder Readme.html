Introduction

libloadorder is a free software library that provides functions for manipulating plugin load order and active status. Its main features are:

  • C frontend.
  • Available as x86 and x64 static and dynamic libraries.
  • Supports TES III: Morrowind, TES IV: Oblivion, TES V: Skyrim, Fallout 3 and Fallout: New Vegas.
  • Supports total conversions which replace any of the supported games' main master files (eg. Nehrim - At Fate's Edge).
  • Provides a uniform interface to the supported games' differing load order systems.
  • Supports the community standard textfile-based load order system for TES V: Skyrim.
  • Supports ghosted plugins.
  • Get/Set the active plugin list.
  • Get/Set the full load order.
  • Get/Set the load order position of an individual plugin.
  • Get/Set the active status of an individual plugin.
  • Uses load order and active plugin list caching to avoid unnecessary disk reads, increasing performance.
  • Free and open source software licensed under the GNU General Public License v3.0.

libloadorder is designed to free modding utility developers from the task of implementing and maintaining their own code for the functionality it provides. It is intended as one component of a suite of libraries for modding utility developers, which has been given the working title of the Common Modding Utility Libraries. The project to develop these libraries is in its infancy, and has not yet settled on a permanent hosting solution, so for the time being libloadorder is hosted on the BOSS repository, where binaries and source code have been made available.

This documentation assumes a familiarity with the basics of load ordering. An introduction to the concepts involved may be found in the Textfile-based Load Order Standard.

Note: libloadorder is not thread safe. Thread safety is a goal, but one that has not yet been achieved. Bear this in mind if using it in a multi-threaded client.

Variable Types

libloadorder uses character strings and integers for information input/output.

  • All strings are unsigned 8-bit character strings (uint8_t*) encoded in UTF-8 and are null-terminated.
  • All codes are unsigned 32-bit integers (uint32_t).
  • All array sizes and positions use the unsigned data type size_t.
  • File paths are case-sensitive if and only if the underlying file system is case-sensitive.

libloadorder also introduces one new structure: typedef struct Game * game_handle;

The game_handle structure abstracts the definition of libloadorder's internal state while still providing type safety across the library's functions.

Memory Management

libloadorder manages the memory of strings and arrays it returns internally, so such strings and arrays should not be deallocated by the client.

A string or array outputted by libloadorder lasts until the next call to a function which outputs data of the same type, eg. a string will last until a function outputting a string is called. DestroyGameHandle will also destroy all outputted strings and arrays.

The one exception to the above is the string outputted by GetLastErrorDetails. This string is allocated independently of all other strings, so lasts until the next call to GetLastErrorDetails or until CleanUpErrorDetails is called. It is not destroyed by DestroyGameHandle.

Library Codes

Return Codes

libloadorder functions use the following uint32_t return codes to signal how a function completes. WARN return codes signify a problem that did not cause the function to exit prematurely, whereas ERROR return codes signify issues that did cause the function to exit prematurely. When a function exits prematurely, a reversal of changes made during its prior execution is attempted.

Return CodeDescription
LIBLO_OK The function completed successfully.
LIBLO_WARN_BAD_FILENAME A plugin filename contains characters that do not have Windows-1251/1252 code points. The plugin cannot be activated.
LIBLO_WARN_LO_MISMATCH This return code can only ever be encountered when using libloadorder with a game that uses the textfile-based load order system. It is only ever returned by CreateGameHandle, and signifies that the load order in the active plugins list file (eg. plugins.txt) is not equivalent to the load order in the full load order file (eg. loadorder.txt). Synchronisation of the two is automatic when load order is managed through libloadorder, but other utilities (eg. the Skyrim launcher) may cause desync to occur. See the Textfile-based Load Order Standard for more information.
LIBLO_ERROR_FILE_READ_FAIL A file could not be read.
LIBLO_ERROR_FILE_WRITE_FAIL A file could not be written to.
LIBLO_ERROR_FILE_RENAME_FAIL A file could not be renamed.
LIBLO_ERROR_PARSE_FAIL There was an error parsing the file.
LIBLO_ERROR_FILE_NOT_UTF8 The specified file is not encoded in UTF-8.
LIBLO_ERROR_FILE_NOT_FOUND The specified file could not be found.
LIBLO_ERROR_TIMESTAMP_READ_FAIL The modification date of a file could not be read.
LIBLO_ERROR_TIMESTAMP_WRITE_FAIL The modification date of a file could not be set.
LIBLO_ERROR_NO_MEM The library was unable to allocate the required memory.
LIBLO_ERROR_INVALID_ARGS Invalid arguments were given for the function.
LIBLO_RETURN_MAX Matches the value of the highest-numbered return code. Provided in case clients wish to incorporate additional return codes in their implementation and desire some method of avoiding value conflicts.

Game Codes

The following uint32_t codes are used by CreateGameHandle to specify which game the handle is for, in cases where distinctions between games are necessary.

Game CodeGame
LIBLO_GAME_TES3 The Elder Scrolls III: Morrowind
LIBLO_GAME_TES4 The Elder Scrolls IV: Oblivion
LIBLO_GAME_TES5 The Elder Scrolls V: Skyrim
LIBLO_GAME_FO3 Fallout 3
LIBLO_GAME_FNV Fallout: New Vegas

Load Order Method Codes

The following codes are used by GetLoadOrderMethod to signify which load order system the game handle is using.

Load Order Method CodeDescription
LIBLO_METHOD_TIMESTAMPThe game handle is using the timestamp-based load order system. Morrowind, Oblivion, Fallout 3 and Fallout: New Vegas all use this system, as does pre-v1.4.26 Skyrim.
LIBLO_METHOD_TEXTFILEThe game handle is using the textfile-based load order system. Skyrim v1.4.26+ uses this system.

Functions

libloadorder provides the functions listed below to clients. Where a function returns a uint32_t value, this is the function's return code.

Error Handling Functions

GetLastErrorDetails - Outputs a string giving the details of the last time an error or warning return code was returned by a function. uint32_t GetLastErrorDetails (uint8_t ** details);
  • details - A pointer to the error details string outputted by the function.
CleanUpErrorDetails - Frees the memory allocated to the last error details string. void CleanUpErrorDetails ();
  • details - A pointer to the error details string outputted by the function.

Version Functions

IsCompatibleVersion - Returns whether this version of libloadorder is compatible with the given version of libloadorder. Abstracts library stability policy away from clients. bool IsCompatibleVersion (const uint32_t bossVersionMajor, const uint32_t bossVersionMinor, const uint32_t bossVersionPatch);
  • versionMajor - The major version number (major.minor.patch) of the library version the client's code was written for.
  • versionMinor - The minor version number (major.minor.patch) of the library version the client's code was written for.
  • versionPatch - The patch version number (major.minor.patch) of the library version the client's code was written for.
GetVersionNums - Returns the version numbers for the loaded version of libloadorder. uint32_t GetVersionNums (uint32_t * versionMajor, uint32_t * versionMinor, uint32_t * versionPatch);
  • versionMajor - The major version number (major.minor.patch) of the library.
  • versionMinor - The minor version number (major.minor.patch) of the library.
  • versionPatch - The patch version number (major.minor.patch) of the library.

Lifecycle Management Functions

CreateGameHandle - Creates a handle for one of the games supported by the library. If the gameId uses the textfile-based load order system, this function also checks if the two load order files are in sync, provided that they both exist. You can create multiple game handles, including for the same game, but remember that libloadorder is not thread safe. uint32_t CreateGameHandle (game_handle * gh, const uint32_t gameId, const uint8_t * gamePath);
  • db - A pointer to the handle created by the function.
  • gameId - A game code specifying which game to create the handle for.
  • gamePath - A string containing the relative or absolute path to the game folder.
DestroyGameHandle - Destroys the given handle, freeing any memory allocated as part of its use. void DestroyGameHandle (game_handle gh);
  • gh - The game handle to be destroyed.
SetNonStandardGameMaster - Sets the game's master file to a given filename, eg. for use with total conversions where the original main master file is replaced with one of a different filename. uint32_t SetNonStandardGameMaster (game_handle gh, const uint8_t * masterFile);
  • gh - The game handle to be destroyed.
  • masterFile - The filename of the replacement main master file.

Plugin Activity Functions

These functions require a game handle. Ghosted plugins can be passed using their ghosted or unghosted filenames.

libloadorder's client interface treats the active plugin list as an unordered set of plugin filenames, so the order of plugins in the inputted and outputted arrays of the functions below holds no significance.

Note that the functions GetActivePlugins and IsPluginActive may output data from an invalid active plugin list, but SetActivePlugins and SetPluginActiveStatus will return with an error if their operation would result in an invalid active plugin list. A valid active plugin list is one that meets all of the following conditions:

  • Contains only installed plugins.
  • Contains no duplicate entries.
  • Contains no more than 255 plugins.
  • If for Skyrim, contains Skyrim.esm.
  • If for Skyrim and Update.esm is installed, contains Update.esm.
GetActivePlugins - Gets an array of all plugins in the active plugin list. uint32_t GetActivePlugins (game_handle gh, uint8_t *** plugins, size_t * numPlugins);
  • gh - The handle the function acts on.
  • plugins - The outputted array of active plugins. If numPlugins is 0, this will be NULL.
  • numPlugins - The size of the outputted plugins array.
SetActivePlugins - Replaces the contents of the active plugin list with the given plugins. uint32_t SetActivePlugins (game_handle gh, uint8_t ** plugins, const size_t numPlugins);
  • gh - The handle the function acts on.
  • plugins - The inputted array of plugins to be made active.
  • numPlugins - The size of the plugins array.
IsPluginActive - Checks to see if the given plugin is in the active plugin list. uint32_t IsPluginActive (game_handle gh, const uint8_t * plugin, bool * result);
  • gh - The handle the function acts on.
  • plugin - A string containing the filename of the plugin to check the active/inactive status of.
  • isActive - A pointer to the boolean output. The output is true if the plugin is active, and false otherwise.
SetPluginActiveStatus - If active is true, activates the given plugin if it is not already active. If active is false, deactivates the plugin if it is not already inactive. If the given plugin is already of the status it is being set to, its status is simply not changed. If activating a plugin that is ghosted, the .ghost extension is removed. uint32_t SetPluginActiveStatus (game_handle gh, const uint8_t * plugin, const bool active);
  • gh - The handle the function acts on.
  • plugin - A string containing the filename of the plugin to change the active/inactive status of.
  • active - Whether to activate or deactivate the plugin. If true, then the plugin is activated if it is not already active. If false, the plugin is deactivated unless it is already inactive.

Plugin Sorting Functions

These functions require a game handle. Ghosted plugins can be passed using their ghosted or unghosted filenames.

libloadorder's client interface uses a zero-based indexing system, so the first plugin in the load order has position 0.

Note that the functions GetLoadOrder, GetPluginLoadOrder and GetIndexedPlugin may output data from an invalid load order, but SetLoadOrder and SetPluginLoadOrder will return with an error if their operation would result in an invalid load order. A valid load order is one that meets all of the following conditions:

  • Contains only installed plugins.
  • Contains no duplicate entries.
  • The first plugin in the load order must be the game's main master file. Use SetNonStandardGameMaster before manipulating load order if the main master file in use does not have the same filename as the original.
  • Loads all master files before all plugin files. Master bit flag value, rather than file extension, is checked.
GetLoadOrderMethod - Returns which method BOSS is using for the load order. uint32_t GetLoadOrderMethod (game_handle gh, uint32_t * method);
  • gh - The handle the function acts on.
  • method - A pointer to the load order method code for the load order system that is being used by the handle gh.
GetLoadOrder - Outputs the current load order. uint32_t GetLoadOrder (game_handle gh, uint8_t *** plugins, size_t * numPlugins);
  • gh - The handle the function acts on.
  • plugins - A pointer to the outputted array of plugins in load order. If numPlugins is 0, this will be NULL. The outputted load order may be invalid.
  • numPlugins - A pointer to the size of the outputted plugins array.
SetLoadOrder - Sets the load order to the passed plugin array, then scans the plugins directory and inserts any plugins not included in the passed array. Plugin files are inserted at the end of the load order, and master files are inserted after the last master file in the load order. The order of plugin insertion is undefined besides the distinction made between master files and plugin files. uint32_t SetLoadOrder (game_handle gh, uint8_t ** plugins, const size_t numPlugins);
  • gh - The handle the function acts on.
  • plugins - The inputted array of plugins in their new load order. This load order must be valid.
  • numPlugins - The size of the plugins array.
GetPluginLoadOrder - Gets the current load order position of the specified plugin. uint32_t GetPluginLoadOrder (game_handle gh, const uint8_t * plugin, size_t * index);
  • gh - The handle the function acts on.
  • plugin - A string containing the filename of the plugin to get the load order position of.
  • index - A pointer to the outputted load order index of the inputted plugin.
SetPluginLoadOrder - Sets the load order position of the specified plugin. uint32_t SetPluginLoadOrder (game_handle gh, const uint8_t * plugin, const size_t index);
  • gh - The handle the function acts on.
  • plugin - A string containing the filename of the plugin to set the load order of.
  • index - The load order index to be set for the inputted plugin. If the index specified is greater than the number of plugins in the load order, the plugin will be inserted at the end of the load order.
GetIndexedPlugin - Gets the filename of the plugin is at the specified load order position. uint32_t GetIndexedPlugin (game_handle gh, const size_t index, uint8_t ** plugin);
  • gh - The handle the function acts on.
  • index - The load order index to be looked at.
  • plugin - A pointer to the outputted filename of the plugin at index in the load order.

Example Usage

The source code of the libloadorder-tester application provides a very simple example of libloadorder usage, both by using the library's C interface directly and by using a C++ wrapper that allows a neater implementation in a C++-based project. The source code for libloadorder-tester is hosted alongside the libloadorder source code.
